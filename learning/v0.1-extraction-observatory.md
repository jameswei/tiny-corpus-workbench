# v0.1 Learning Module: Extraction Observatory

This lesson explores what can go wrong before a document reaches chunking,
retrieval, or generation. You will run two extraction paths over the same
source, inspect their evidence, and test what the verifier can and cannot
promise.

This lesson has a narrow scope. v0.1 observes extraction. It does not select a
better output, diagnose a problem, or repair a document.

## Learning objectives

By the end, you can:

1. explain why extraction output must remain inspectable
2. distinguish a source, private snapshot, extraction view, and observation
   manifest
3. compare Docling and MarkItDown without treating a metric delta as a quality
   score
4. explain application-immutable and tamper-evident in precise terms
5. interpret `VERIFIED`, `INTEGRITY_MISMATCH`, and `BROKEN`
6. separate historical artifact integrity from current source/model freshness
7. state the trusted-local threat model and its deliberate limits

Suggested time: 60–90 minutes. Commands assume the repository root and
CPython 3.12.

## 1. Build the mental model

An observation follows this path:

```text
original source
     |
     | validate, open once, copy bytes
     v
private source snapshot
     |                    |
     v                    v
  Docling             MarkItDown
     |                    |
     +---------+----------+
               |
               v
       staged observation
               |
       schema + inventory
          verification
               |
               v
       atomic publication
```

Both extractors must process identical bytes. Therefore, the application gives
each extractor the same private snapshot. Without this snapshot, a source
change could give each extractor a different input.

The application verifies the staged files, schemas, hashes, and internal
relationships. It publishes the directory only after this verification. It
does not repair or overwrite a published run.

### Checkpoint

Explain why two independent source reads can cause an incorrect comparison.
Explain how one captured snapshot prevents this problem.

## 2. Observe a source without PDF models

Install the exact locked environment:

```bash
uv sync --frozen --python 3.12
```

Start with Markdown, which does not require Docling's PDF model inventory:

```bash
uv run --frozen tcw observe fixtures/golden/policy-memo.md
```

The command prints one compact JSON line. The line contains the status, run ID,
and `manifest.json` path. Set `RUN_DIR` to the parent directory of the manifest:

```bash
RUN_DIR="build/extraction-observatory/<source-key>/<run-id>"
```

Inspect the directory without changing it:

```bash
find "$RUN_DIR" -type f -print | sort
uv run --frozen python -m json.tool "$RUN_DIR/manifest.json" | less
uv run --frozen python -m json.tool "$RUN_DIR/comparison.json" | less
```

Expected evidence:

```text
manifest.json
comparison.json
docling/document.json
docling/document.md
markitdown/document.md
```

`docling/document.json` is the lossless canonical extraction artifact.
Markdown is a derived view for reading and comparison.

### Questions to investigate

- Which source size and SHA-256 does the manifest record?
- Which exact runtime dependencies does the manifest name?
- Which extractor configuration disables remote or interpretive behavior?
- Which artifacts carry `application_immutable: true`?
- Why does the manifest record `models.required = false` for this Markdown
  observation?

## 3. Compare views without declaring a winner

Read the two Markdown views:

```bash
less "$RUN_DIR/docling/document.md"
less "$RUN_DIR/markitdown/document.md"
```

Then inspect `comparison.json`. It reports deterministic facts such as:

- normalized content hashes
- bytes, characters, and lines
- heading and list-item counts
- pipe-table row counts
- visible URL counts
- required fixture-anchor presence

A delta is always Docling minus MarkItDown. A nonzero delta identifies a
difference to inspect. It does not identify the correct extractor. Equal counts
do not prove semantic equivalence or a correct reading order.

### Exercise: form an evidence statement

Write one sentence using this pattern:

> The views differ in ___ according to ___. Manual inspection shows ___. v0.1
> does not infer ___.

Avoid words such as *better*, *accurate*, or *broken* unless the evidence you
cite actually supports that conclusion.

## 4. Add the PDF model boundary

PDF extraction requires an explicit local model inventory. Download the models
during setup. You can use network access during this setup step.

```bash
uv run --frozen docling-tools models download layout tableformer \
  --output-dir .cache/docling/models
```

Now observe a PDF:

```bash
uv run --frozen tcw observe fixtures/golden/policy-memo.pdf \
  --docling-artifacts .cache/docling/models
```

Extraction is offline during an observation. The application hashes the
complete sorted model inventory before and after extraction. It stops
publication if a model file changes, appears, disappears, is unreadable, or is
a symbolic link.

### Checkpoint

Why does the application record the model inventory as provenance? Why does the
repository not contain the model files?

## 5. Verify an untouched observation

Run the self-contained verifier:

```bash
uv run --frozen tcw verify "$RUN_DIR"
```

An untouched run must return exit `0`. Its JSON report must show artifact status
`VERIFIED`.

For v0.1, `VERIFIED` means, given the manifest as the local record:

1. supported structures and schemas are valid
2. persisted artifacts match their recorded kinds, sizes, and hashes
3. the semantic relationships explicitly enumerated by the contract agree

It does not authenticate the author, timestamp, original source, or runtime
environment.

## 6. Make corruption visible—safely

Do not change the original observation. Copy it to a temporary directory. Keep
the run directory name unchanged.

```bash
LAB_ROOT="$(mktemp -d)"
cp -R "$RUN_DIR" "$LAB_ROOT/"
LAB_RUN="$LAB_ROOT/$(basename "$RUN_DIR")"
```

Change one copied artifact:

```bash
printf '\nlearner edit\n' >> "$LAB_RUN/markitdown/document.md"
uv run --frozen tcw verify "$LAB_RUN"
```

Expected result: The command returns exit `5`. The report has an
`INTEGRITY_MISMATCH` or `BROKEN` status. The evidence identifies the
inconsistent path. The verifier does not change the copied run.

This behavior is tamper-evident. The verifier can detect an ordinary
uncoordinated change. This behavior is not tamper-proof. The filesystem does
not prevent another process or user from changing bytes.

## 7. Separate historical integrity from current freshness

The observation directory contains historical evidence. The current source and
PDF model inventory are separate. You can compare them with the recorded
provenance.

Try the source advisory with a disposable source copy:

```bash
SOURCE_COPY="$LAB_ROOT/policy-memo.md"
cp fixtures/golden/policy-memo.md "$SOURCE_COPY"
uv run --frozen tcw observe "$SOURCE_COPY" \
  --output-root "$LAB_ROOT/observations"
```

Get the new run directory from the JSON output. Set `FRESH_RUN` to this path.
Then, change only the current source:

```bash
FRESH_RUN="$LAB_ROOT/observations/<source-key>/<run-id>"
printf '\nCurrent source changed after observation.\n' >> "$SOURCE_COPY"
uv run --frozen tcw verify "$FRESH_RUN" --source "$SOURCE_COPY"
```

Expected result:

- artifact integrity remains `VERIFIED`
- source state becomes `CHANGED`
- verifier exit remains `0`

The current source changed, but the historical observation did not change. The
source advisory answers a different question.

## 8. Know the limits of `VERIFIED`

Use this table to predict v0.1 behavior:

| Scenario | Expected interpretation |
| --- | --- |
| One artifact's bytes change | Detectable mismatch or broken relationship |
| A referenced file disappears | Detectable mismatch or broken relationship |
| An unexpected file appears | Detectable inventory violation |
| Current source changes later | Advisory `CHANGED`; historical integrity unchanged |
| A person replaces one valid timestamp with another | The record can remain structurally valid; v0.1 does not promise authenticity |
| Manifest and every artifact are coherently rewritten | Outside the self-contained local-hash guarantee |
| Same-user process injects into the running verifier | Outside the trusted-local threat model |
| A person disputes a cryptographic signer identity | v0.1 does not represent this case |

The key distinction is:

```text
integrity: do the recorded relationships still agree?
authenticity: can an external party trust who or what created the record?
```

v0.1 checks integrity in a trusted local environment. It does not claim
authenticity.

## 10. Knowledge check

1. Why must both extractors consume one source snapshot?
2. Why is `comparison.json` descriptive rather than evaluative?
3. What is the difference between `INTEGRITY_MISMATCH` and `BROKEN`?
4. Why can a changed source advisory coexist with artifact `VERIFIED`?
5. Why can local hashes provide tamper evidence without providing
   authenticity?
6. Why does rebuilding create a new run instead of repairing the old one?

### Answer guide

1. It prevents mixed inputs when the original changes during extraction.
2. Counts and hashes identify differences. They do not establish correctness.
3. `INTEGRITY_MISMATCH` means that recorded bytes or kinds differ.
   `BROKEN` means that a structure or specified semantic relationship is
   invalid.
4. `VERIFIED` describes a historical run. The source advisory compares a
   current source with that history.
5. A person can rewrite the local record and calculate new hashes. No external
   identity or anchor exists.
6. A new run preserves the old evidence and history for inspection.

## 11. Bridge to v0.2

v0.1 answers: *What did each extractor produce, and is this observation still
internally consistent?*

v0.2 will ask: *Which concrete document-quality problems can diagnostic rules
identify with repeatable evidence?*

Before proposing a diagnosis rule, identify:

- the exact observable symptom
- the affected document item or artifact path
- deterministic evidence
- likely false positives
- whether diagnosis alone authorizes a change—it must not

Keep observation, diagnosis, and refinement as separate decisions. Do not
combine them into one unclear cleanup step.
