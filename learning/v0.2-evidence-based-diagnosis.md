# Evidence-Based Diagnosis

## Goal

Learn how fixed rules produce evidence without changing a document.

## Key terms

A diagnosis is a read-only analysis of one canonical document.

A finding is one rule result with stable references and evidence.

A diagnosis ID binds the observation, canonical document, and ruleset.

## Prepare an observation

1. Observe the short diagnostic fixture.

   ```bash
   uv run --frozen tcw observe fixtures/diagnosis/v0.2/short-note.md
   ```

2. Copy the observation directory from the command output.

3. Set it as `OBSERVATION_DIRECTORY` in the next commands.

## Create a diagnosis

1. Run diagnosis.

   ```bash
   uv run --frozen tcw diagnose OBSERVATION_DIRECTORY
   ```

2. Open `findings.json`.

3. Find rule `TCW-D002`.

4. Check its character count and document reference.

5. Open `report.md`.

6. Confirm that the report has the same finding.

The manifest has a unique run ID. The finding set and report are deterministic.

## Verify the evidence

1. Run independent diagnosis verification.

   ```bash
   uv run --frozen tcw verify-diagnosis DIAGNOSIS_DIRECTORY
   ```

2. Confirm that `artifact_integrity.status` is `VERIFIED`.

3. Rerun the rules against the observation.

   ```bash
   uv run --frozen tcw verify-diagnosis DIAGNOSIS_DIRECTORY \
     --observation OBSERVATION_DIRECTORY
   ```

4. Confirm that both optional states are `MATCH`.

## Inspect structural findings

1. Observe `fixtures/diagnosis/v0.2/structural-traps.md`.

2. Diagnose the new observation.

3. Find rules D003, D004, and D005.

4. Compare their evidence shapes.

D003 uses offsets. D004 uses references, a count, and a text hash. D005 uses
heading levels and the previous heading reference.

## Perform a safe integrity experiment

1. Copy one diagnosis directory.

2. Edit `report.md` in the copy.

3. Verify the copied diagnosis.

4. Confirm exit code `5`.

5. Confirm that `artifact_integrity.status` is `INTEGRITY_MISMATCH`.

Do not edit the original publication.

## Knowledge check

1. Why can MarkItDown output not create a v0.2 finding?
2. Why is `NO_FINDINGS` not proof of correctness?
3. Which bytes make a diagnosis ID stable?
4. Why are optional observation states advisory?
5. Does a finding authorize a repair?

## Expected answers

1. The canonical Docling document is the only diagnostic content input.
2. The fixed rules cover only defined mechanical conditions.
3. Observation identity, manifest hash, document hash, and ruleset define it.
4. A diagnosis remains an immutable record when its source copy is unavailable.
5. No. Diagnosis does not authorize mutation.
