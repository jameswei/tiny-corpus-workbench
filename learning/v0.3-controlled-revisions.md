# Learn v0.3: Controlled Revisions

Estimated study time: 60 minutes.

## Learning goals

After this lesson, you can:

- explain why a finding does not approve a change
- draft and resolve one refinement decision
- compare an approval with a rejection
- inspect transformation history
- verify forward and inverse operations
- create a second revision from a new diagnosis

## Prepare the inputs

Use CPython 3.12 and the locked environment:

```bash
uv sync --frozen --python 3.12
```

Start with the v0.3 fixtures:

```bash
uv run --frozen tcw observe fixtures/refinement/v0.3/whitespace-cleanup.md
uv run --frozen tcw observe fixtures/refinement/v0.3/line-end-hyphenation.docx
```

Keep the manifest paths from both commands. The observation directory contains
the printed manifest.

## Approve whitespace normalization

1. Diagnose the whitespace observation.

```bash
uv run --frozen tcw diagnose OBSERVATION_DIRECTORY
```

2. Open `findings.json`.

3. Copy the `finding_id` for `TCW-D009`.

4. Create a draft.

```bash
uv run --frozen tcw draft-refinement DIAGNOSIS_DIRECTORY \
  --finding FINDING_ID \
  --base OBSERVATION_DIRECTORY \
  --output whitespace-decision.json
```

5. Edit only the `decision` object.

6. Set `state` to `APPROVED`.

7. Set `decided_by` to your actor label.

8. Keep `note` as `null` or add a short note.

9. Resolve the draft.

```bash
uv run --frozen tcw resolve-refinement whitespace-decision.json \
  --diagnosis DIAGNOSIS_DIRECTORY \
  --base OBSERVATION_DIRECTORY
```

10. Verify the result.

```bash
uv run --frozen tcw verify-refinement REFINEMENT_DIRECTORY \
  --diagnosis DIAGNOSIS_DIRECTORY \
  --base OBSERVATION_DIRECTORY
```

Confirm that derivation and reversibility are `MATCH`.

## Approve a dehyphenation

Repeat the diagnosis and draft steps for the line-end hyphenation observation.
Select `TCW-D010`.

Set the decision state to `APPROVED`. Add an actor label. Resolve and verify
the draft with the line-end hyphenation observation as the base.

Confirm these results:

- the prepared text joins `Inter-` and `operable`
- derivation is `MATCH`
- reversibility is `MATCH`

## Reject a dehyphenation

Create a new draft for the same `TCW-D010` finding and original observation.
Set the decision state to `REJECTED`. Add an actor label. Resolve the draft.

Confirm these results:

- the command exits `0`
- the manifest status is `REJECTED`
- `revision_id` is null
- no `prepared/`, `transformation.json`, or `history.json` exists
- derivation and reversibility are `NOT_APPLICABLE`

## Create a revision chain

1. Diagnose an approved refinement directory.

2. Select one supported finding from the new diagnosis.

3. Use the approved refinement directory as `--base`.

4. Approve and resolve the second draft.

5. Open `history.json`.

Confirm that the child history contains the parent transformation first and
the new transformation second.

## Perform safe tamper experiments

Work only on copies.

1. Copy an applied refinement directory.

2. Edit `report.md` in the copy.

3. Run `verify-refinement`.

4. Confirm exit `5`.

Repeat the experiment with `prepared/document.json`,
`transformation.json`, and `history.json`.

Do not change the original observation, diagnosis, decision, or refinement
publication.

## Knowledge check

1. Does a diagnosis finding approve a change?
2. Which draft fields can you edit?
3. What does a rejected decision publish?
4. Why must you diagnose a revision again before its successor?
5. What does reversibility `MATCH` prove?

## Expected answers

1. No. A person must resolve the decision.
2. `decision.state`, `decision.decided_by`, and `decision.note`.
3. An immutable manifest, decision, and report. It publishes no revision.
4. Each draft must use a current finding for its exact base document.
5. The recorded inverse operation reproduces the base document bytes.
