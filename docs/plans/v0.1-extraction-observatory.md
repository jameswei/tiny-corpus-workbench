# Milestone Plan: v0.1 — Extraction Observatory

**Slug:** `extraction-observatory`
**Status:** Accepted on 2026-07-21; local-integrity amendment accepted on
2026-07-21

## 1. Outcome

Build a local, service-free CLI that runs Docling and MarkItDown against the
same source and publishes an application-immutable, locally tamper-evident
observation containing:

- source identity, media type, size, and SHA-256
- lossless `DoclingDocument` JSON
- Markdown serialized by Docling
- Markdown produced by MarkItDown
- a project-owned preparation manifest
- a deterministic descriptive comparison summary

The milestone teaches how two extraction views preserve or lose observable
content and structure. It will not diagnose quality, rank extractors,
recommend repairs, or claim semantic equivalence.

## 2. Boundaries

In scope:

- one local source per command
- PDF, DOCX, Markdown, and strict UTF-8 plain text
- Docling and MarkItDown adapters
- exactly twelve controlled fixtures
- an original source the application never mutates
- one private byte snapshot used by both extractors
- application-immutable, locally tamper-evident published runs
- read-only verification of published observations
- dependency, schema, configuration, and model provenance
- offline observation after explicit model prefetch
- CLI, schemas, tests, and documentation

Out of scope:

- findings, severity, diagnosis, or repair recommendations
- refiners, approvals, transformations, or prepared revisions
- OCR-heavy or scanned documents; OCR remains disabled
- batch/corpus commands, directories, stdin, URLs, connectors, or crawling
- services, HTTP APIs, browser interfaces, or corpus reports
- plugins, cloud extractors, LLM calls, or silent fallback conversion
- chunking, embeddings, indexing, retrieval, generation, and RAG evaluation

- Reference and acceptance runtime: CPython 3.12
- Preserve package declaration `requires-python = ">=3.11"`
- Project version becomes `0.1.0`
- Exact runtime pins:
  - `docling==2.113.0`
  - `markitdown[pdf,docx]==0.1.6`
- Exact fixture-generation pins:
  - `reportlab==5.0.0`
  - `python-docx==1.2.0`
- Exact runtime schema validator:
  - `jsonschema==4.26.0`
- Commit the complete `uv.lock`
- Documented commands use `uv sync --frozen` and `uv run --frozen`
- Runtime code uses standard-library `argparse`; no CLI or runtime validation

`document.json` is the unmodified `save_as_json()` output. Compatibility is
promised only with the exact locked Docling/docling-core environment that
created it. Dependency upgrades create new observations and never rewrite old
artifacts.

## 5. Mandatory compatibility spike

After the accepted plan is committed, implementation begins with only the
dependency/lock setup and a compatibility test.

Using Python 3.12 and the exact pins, it must prove:

1. Docling converts one minimal PDF, DOCX, Markdown, and text input.
2. Each Docling conversion emits non-empty JSON and Markdown.
3. Each JSON artifact reloads under the locked Docling environment.
4. MarkItDown emits non-empty Markdown for all four formats.
5. Prefetched `layout` and `tableformer` files are sufficient for offline PDF
   conversion.
6. OCR, plugins, remote services, LLM calls, and observation-time network
   access remain disabled.
7. Exact package versions and the emitted Docling schema identity/version can
   be recorded.

Pass requires eight non-empty extraction views, four Docling JSON round trips,
and no attempted network access during observation.

Failure stops implementation and returns to the owner. The builder must not
add pass-through output, enable remote behavior, broaden dependency extras, or
weaken the two-extractor contract.

## 6. CLI contract

```text
tcw observe SOURCE
  [--output-root PATH]
  [--docling-artifacts PATH]
```

Defaults:

- `--output-root build/extraction-observatory`
- `--docling-artifacts .cache/docling/models`

Validation:

- `SOURCE` must be one regular local file.
- Directories, URLs, stdin, FIFOs, and unsupported suffixes are rejected.
- Accepted mappings:
  - `.pdf` -> `application/pdf`
  - `.docx` -> Office Open XML document media type
  - `.md` -> `text/markdown`
  - `.txt` -> `text/plain`
- PDF magic, required DOCX ZIP members, strict UTF-8 decoding, and NUL absence
  are checked.
- Extension/content mismatches are input errors.

Execution and publication:

- Both extractors are attempted independently after validation.
- Source SHA-256 is checked before and after extraction.
- Output is built in a private staging directory and published through one
  atomic directory rename.
- Published artifacts are never modified, repaired, or reused by the
  application.
- A source mutation or integrity failure discards staging.
- Valid partial and failed observations are published with evidence and error
  records.
- `comparison.json` is always published for a validated source: `COMPLETE`,
  `INCOMPLETE`, or `NOT_AVAILABLE`.
- A crash before a valid manifest can be completed leaves no published run.

Successful observation attempts emit exactly one sorted-key JSON line to
stdout:

```json
{"manifest":".../manifest.json","run_id":"...","status":"SUCCESS"}
```

Diagnostics go to stderr. Validation failures produce no stdout.

Exit codes:

| Code | Meaning |
| --- | --- |
| `0` | Both extractors completed successfully |
| `1` | Unexpected top-level internal failure |
| `2` | Usage, validation, or unsupported-media error |
| `3` | Partial success: one extractor failed or Docling reported partial success |
| `4` | Neither extractor produced a usable view |
| `5` | Source mutation, publication conflict, or artifact-integrity failure |
| `6` | Locked runtime, dependency, or required model artifacts unavailable/incompatible |

## 7. Fixture contract

The fixtures are fictional, project-authored data dedicated under `CC0-1.0`,
separate from the repository's MIT-licensed code.

Three authoritative authored JSON files generate exactly four representations
each:

| Family | Identity and content |
| --- | --- |
| `policy-memo` | `PM-2026-04`, “Remote Equipment Return Policy,” dated `2026-07-15`; four sections, three bullets, three numbered steps, one 3x3 table, and `https://example.invalid/equipment-returns` |
| `meeting-minutes` | `MM-2026-07-18`, “Atlas Migration Weekly Minutes,” dated `2026-07-18`; five sections, bullets, numbered follow-ups, one 3x3 table, and `https://example.invalid/atlas-minutes` |
| `release-notice` | `RN-2.4.0`, “Northstar 2.4.0 Release Notice,” dated `2026-07-20`; four sections, bullets, upgrade steps, one 3x3 table, and `https://example.invalid/northstar-2-4-0` |

Representations:

- Markdown: ATX headings, lists, and pipe table
- Text: labelled headings and fixed-width table
- DOCX: native headings, lists, table, and hyperlink
- PDF: born-digital, built-in fonts, fixed metadata, and visible hyperlink
- No images, scans, macros, external resources, or real personal data

Generation must be byte-deterministic:

- UTF-8, LF, and one terminal newline for Markdown/text
- ReportLab invariant mode, fixed metadata, built-in fonts, stable draw order
- Fixed OOXML timestamps and sorted DOCX ZIP members with fixed ZIP metadata
- `generate_fixtures.py --check` generates twice in independent temporary
  directories and requires byte equality
- Normal tests never rewrite committed fixtures or their registry

`fixtures.json` uses `tcw.fixture-registry/v0.1`, contains exactly twelve
entries sorted by ID, and records for each fixture:

- fixture ID, family, format, path, media type, size, and SHA-256
- authoritative authored-source ID, path, and hash
- generator name/schema and lockfile hash
- ownership and `CC0-1.0` license
- required document ID, date, and URL anchors
- expected Docling table count: one for PDF/DOCX/Markdown and zero for text

Verification rejects missing/unregistered files, duplicates, unexpected
counts, and hash or metadata mismatches.

## 8. Manifest contract

`manifest.json` uses JSON Schema Draft 2020-12 and:

```text
schema_version = tcw.preparation-manifest/v0.1
milestone = v0.1
```

Required top-level fields:

- `run_id`, `observation_id`, `created_at`, and overall status
- source identity and fixture association
- runtime/dependency/lockfile versions
- fixed extractor configurations
- Docling document schema identity and exact-lock compatibility statement
- model artifact requirement and inventory
- exactly one Docling and one MarkItDown result
- artifact paths, roles, media types, sizes, hashes, and immutability marker
- comparison status and sanitized stable errors

`observation_id` is a SHA-256 over a compact, sorted-key UTF-8 JSON object
containing source identity/media type, extractor/config versions, lock hash,
and model-inventory hash.

Extractor status is:

```text
SUCCESS | PARTIAL_SUCCESS | FAILED
```

Each extractor records version, timing, emitted artifacts, upstream status
where applicable, and either `null` or a stable error object. Stable error
codes include:

- `MODEL_ARTIFACTS_MISSING`
- `MODEL_ARTIFACTS_INVALID`
- `DOCLING_CONVERSION_FAILED`
- `DOCLING_SERIALIZATION_FAILED`
- `MARKITDOWN_CONVERSION_FAILED`
- `INTERNAL_ERROR`

Error messages are single-line, sanitized, and exclude tracebacks and source
content. Unknown fields are rejected by the v0.1 schemas and
schema-conformance tests.

## 9. Comparison contract

`comparison.json` uses:

```text
schema_version = tcw.comparison-summary/v0.1
normalization_algorithm = tcw-markdown-normalize-v1
```

Normalization is exactly:

1. Strict UTF-8 decode.
2. Convert CRLF and CR to LF.
3. Apply Unicode NFC normalization.
4. Remove trailing spaces and tabs per line.
5. Remove leading and trailing blank lines.
6. Append one final LF when non-empty.

Metrics are computed over normalized text. Each available view records:

- artifact and normalized hashes
- presence of registered ID/date/URL anchors
- bytes and characters
- non-whitespace characters
- lines and non-empty lines
- ATX headings
- unordered and ordered list items
- pipe-table rows
- visible URLs

When both views exist, `deltas` contains `Docling - MarkItDown` for every
numeric metric plus `normalized_equal`. There is no fuzzy score, semantic
similarity, ranking, or quality label.

Serialization uses fixed ordering, compact separators, UTF-8, and one terminal
newline. For the same locked inputs and model inventory, `comparison.json`
must be byte-identical across isolated runs.

## 13. Accepted local-integrity amendment

This amendment was accepted on 2026-07-21 after milestone review clarified
that the original word “immutable” did not define a same-user threat model.
Where this section conflicts with an earlier clause, this section controls.
The CLI binary remains `tcw`.

### 13.1 Integrity terminology

“Application-immutable” means `tcw` never overwrites, repairs, deletes, or
reuses a published run. It does not mean the local filesystem prevents its
owner from editing files.

“Locally tamper-evident” means hashes, schemas, exclusive publication, and the
read-only verifier detect ordinary corruption and uncoordinated changes. They
do not establish cryptographic authenticity, attribution, trusted timestamps,
or protection from coordinated artifact-and-manifest rewriting.

Artifact descriptors use:

```json
{"application_immutable":true}
```

instead of `immutable`.

### 13.2 Trusted-local threat model

v0.1 assumes a trusted local user, operating system, Python process, and
filesystem implementation. Users should avoid deliberately modifying the
Docling model cache, staging tree, output tree, or running process during an
observation.

In scope are ordinary failures and neutral concurrency:

- a source changing while its private snapshot is captured
- a source changing after snapshot capture
- a PDF model inventory changing during extraction
- concurrent `tcw observe` processes
- destination collisions
- missing, truncated, extra, replaced, edited, or symlinked observation files
- invalid schemas or inconsistent internal references
- broken or incompatible extractor installations

Explicitly excluded are:

- a same-user actor deliberately timing a staging mutation after final
  verification and before publication
- process injection, debugger manipulation, or replacement of running code
- malicious filesystem or operating-system behavior
- coordinated rewriting of a manifest and all referenced artifacts
- signing, trusted external timestamps, remote attestations, ACL enforcement,
  sandboxes, ledgers, or cryptographic attribution

Final staged verification and exclusive no-replace publication remain
required. Their promise is limited to application behavior and ordinary
detected changes under this threat model.

### 13.3 Extractor runtime preflight

Before source capture or staging, `tcw observe` checks:

- installed package metadata matches the locked versions
- both adapter modules import successfully
- each adapter resolves the Docling or MarkItDown symbols and APIs it uses

Broken imports, missing symbols, and incompatible locked APIs are runtime
contract failures. They emit one sanitized stderr message, publish nothing,
and exit `6`.

Conversion or serialization failures after a successful preflight retain
their extraction exit behavior. Initial missing or invalid PDF models retain
their recorded model-error behavior and exit `6`.

### 13.4 Private source snapshot

`tcw observe` uses this capture sequence:

1. Reject a symlink or non-regular source.
2. Open the source once without following symlinks where the platform supports
   that flag.
3. Record file-descriptor identity and metadata before copying.
4. Copy and hash the bytes into an owner-only private temporary snapshot using
   the original basename and suffix.
5. Recheck the open source descriptor after copying. A size, identity, or
   metadata change during capture is an integrity failure and exits `5`.
6. Validate media content against the snapshot and derive manifest source
   identity from its bytes.
7. Run both Docling and MarkItDown exclusively against that same snapshot.
8. Remove the snapshot before publication. Cleanup failure prevents
   publication and exits `5`.

Snapshot cleanup occurs on success, extractor failure, runtime failure,
integrity failure, and unexpected exceptions. The snapshot is temporary
processing state and is not published.

Once a valid snapshot has been captured, later changes to the original source
do not fail the observation and cannot mix extractor inputs. Such changes are
reported only by an opt-in verifier freshness check.

Concurrent observations use independent snapshots, staging directories, and
run IDs. A genuine destination collision never replaces the winner.

### 13.5 PDF model stability

For PDF observations:

1. Compute and validate the complete sorted model inventory before extraction.
2. Run Docling against that explicit model root.
3. Recompute the inventory after extraction and immediately before final
   staged verification.
4. Require identical canonical inventory descriptors and hashes.

An initially missing or invalid required inventory is exit `6`. A model file
added, removed, replaced, resized, rehashed, changed into a symlink, or made
unreadable between inventories is an integrity failure: discard staging,
publish nothing, and exit `5`.

Non-PDF observations record `models.required = false` and do not inventory the
model directory. A model change after the final inventory does not alter the
published extraction artifacts and is only an optional verifier advisory.

### 13.6 Manifest and comparison amendments

The manifest gains:

- `source.capture_method`, constant `private-byte-snapshot-v1`
- `comparison.size` and `comparison.sha256`
- `application_immutable: true` on every persisted artifact descriptor

The comparison file is part of the manifest’s persisted size/hash inventory.
`comparison.json` gains required `observation_id`, matching the manifest.
Manifest status, extractor status, comparison status, artifact presence, and
references must be mutually consistent.

`observation_id` uses the captured source identity and pre-extraction model
inventory. `manifest.json` cannot contain a trustworthy hash of itself; its
protection is schema validation and internal consistency under the
trusted-local threat model. The verifier must not claim it can detect a
coordinated rewrite of the manifest and every artifact.

Add:

```text
src/tiny_corpus_workbench/verification.py
src/tiny_corpus_workbench/schemas/verification-result-v0.1.schema.json
tests/unit/test_verification.py
```

### 13.7 Read-only verifier

Add:

```text
tcw verify OBSERVATION_DIRECTORY
  [--source SOURCE]
  [--docling-artifacts PATH]
```

Without optional arguments, verification is self-contained and reads only the
observation directory and bundled schemas. Current source and model checks are
opt-in; historical paths recorded in the manifest are never followed
automatically.

`--source SOURCE` compares a current regular, non-symlink file with the
captured size and SHA-256. `--docling-artifacts PATH` compares a current model
inventory with the recorded PDF inventory. A missing optional target is an
advisory `MISSING` state, not a CLI usage error.

The command never imports extractor adapters, invokes extraction, downloads
models, repairs files, changes permissions, rewrites metadata, creates
sidecars, deletes files, or intentionally changes access or modification
times.

The verifier must:

- reject a symlink observation root
- load `manifest.json` and validate its exact supported schema
- recompute `observation_id`
- require the directory basename to equal `manifest.run_id`
- derive the exact expected file/directory set from the manifest
- reject unsafe, escaping, absolute, or duplicate paths
- reject missing/unexpected paths and invalid file kinds
- verify every persisted artifact size and SHA-256
- validate `comparison.json` and its source, observation ID, status, view
  hashes, and manifest references
- verify overall/extractor/comparison status and artifact consistency
- recompute the recorded model inventory hash from its sorted descriptors

### 13.8 Verification result and exits

Stdout is exactly one compact, sorted-key JSON line conforming to
`tcw.verification-result/v0.1` and containing:

```json
{
  "artifact_integrity":{"issues":[],"status":"VERIFIED"},
  "model_state":{"status":"NOT_CHECKED"},
  "observation_directory":"/absolute/path/to/run",
  "schema_version":"tcw.verification-result/v0.1",
  "source_state":{"status":"NOT_CHECKED"}
}
```

Artifact states:

```text
VERIFIED | INTEGRITY_MISMATCH | BROKEN
```

Source states:

```text
MATCH | CHANGED | MISSING | ERROR | NOT_CHECKED
```

Model states:

```text
MATCH | CHANGED | MISSING | ERROR | NOT_CHECKED | NOT_APPLICABLE
```

For v0.1, `VERIFIED` means that, given the observation manifest as the local
record of the run:

1. the observation is structurally valid
2. its persisted artifacts match that record
3. all explicitly derivable relationships are semantically consistent

Structural validity covers supported schemas, required/unknown fields, types,
enums, safe paths, RFC 3339 timestamp syntax, nonnegative durations, and
sanitized single-line errors.

Artifact integrity covers the exact expected file/directory inventory,
regular non-symlink kinds, persisted sizes, and SHA-256 hashes.

Semantic consistency covers the relationships explicitly enumerated by this
plan: runtime and extractor versions, Docling schema identity derived from
`document.json`, upstream/extractor/overall/comparison statuses, PDF model
applicability and inventory, artifact roles/paths/media types, comparison
identity/views/references, and observation ID recomputation.

`INTEGRITY_MISMATCH` means the observation is interpretable but bytes, kinds,
sizes, hashes, or paths differ. `BROKEN` means structural validity or an
enumerated semantic relationship fails. `BROKEN` takes precedence over
`INTEGRITY_MISMATCH`.

`VERIFIED` does not authenticate who created the observation, when it was
created, whether the source was genuine, or whether the execution environment
was trustworthy. It cannot distinguish an original non-derivable metadata
value from a different schema-valid value, and it cannot detect a coordinated
rewrite of the complete local record. Valid-looking changes to non-derivable
metadata are outside v0.1 verification acceptance.

Source and model states are advisory and never change artifact integrity.
Issues have stable uppercase codes, observation-relative paths or `null`, and
sanitized single-line messages. Required codes include:

```text
MANIFEST_MISSING
MANIFEST_INVALID
SCHEMA_UNSUPPORTED
SCHEMA_INVALID
RUN_ID_MISMATCH
OBSERVATION_ID_MISMATCH
UNSAFE_REFERENCE
REFERENCE_MISMATCH
STATUS_MISMATCH
FILE_MISSING
FILE_UNEXPECTED
DIRECTORY_UNEXPECTED
FILE_KIND_INVALID
SIZE_MISMATCH
HASH_MISMATCH
COMPARISON_INVALID
```

A completed report, including mismatch or broken results, writes the JSON line
to stdout and nothing to stderr.

| Code | `tcw verify` meaning |
| --- | --- |
| `0` | Artifact status `VERIFIED`; advisories may differ |
| `1` | Unexpected verifier failure |
| `2` | Invalid CLI use or invalid observation-directory argument |
| `5` | Artifact status `INTEGRITY_MISMATCH` or `BROKEN` |
| `6` | Bundled verifier/schema runtime unavailable or incompatible |

Codes `3` and `4` are not used by `tcw verify`.

### 13.9 Re-observation, not repair

A rebuild is another `tcw observe SOURCE` invocation producing a new run. A
questionable old run remains inspectable and is never repaired, overwritten,
quarantined, or deleted by `tcw`. Users may retain it, exclude it, compare it
with a new run, or re-observe a changed source.

### 13.10 Required amendment tests

Source snapshot tests cover:

- both adapters receive the same snapshot path and bytes, never the original
- source modification during capture fails cleanly
- changes after capture cannot mix extractor inputs
- source symlinks are rejected
- snapshots are removed on every success/failure path
- cleanup failure prevents publication

Model/runtime tests cover:

- unchanged pre/post PDF inventories succeed
- additions, removals, byte changes, replacements, and new symlinks exit `5`
- initial missing/invalid models retain exit `6`
- non-PDF runs do not inventory models
- broken adapter imports and API incompatibilities exit `6`
- conversion failures after preflight retain extraction behavior

Neutral concurrency/publication tests cover:

- concurrent observations use independent snapshot, staging, and run paths
- two ordinary concurrent runs may publish distinct observations
- a real run-ID collision never replaces an existing destination
- ordinary staged corruption detected before publication aborts cleanly
- deliberately timed same-user mutation after final verification is outside
  scope and is not a blocking acceptance test

Verifier tests cover:

- untouched success, partial, and failed observations
- missing, extra, edited, truncated, symlinked, and special-kind paths
- invalid manifest/comparison JSON and schema/reference inconsistencies
- recomputed observation and run IDs
- no mutation of bytes, permissions, or modification times
- operation when extractor imports are broken
- every source/model advisory state
- changed/missing current source or models with artifact status still verified
- exact result schema and stdout/stderr/exit behavior

### 13.11 Amended acceptance criteria

In addition to the unchanged extraction and fixture criteria, acceptance
requires:

- `tcw observe` never writes to the original source
- both extractors consume one validated private snapshot
- PDF model inventory is identical before and after extraction
- publication is final-verified, atomic, exclusive, and no-replace
- `tcw` never overwrites, repairs, deletes, or reuses a published run
- `tcw verify NEW_RUN` returns `VERIFIED` and exit `0`
- an uncoordinated artifact change produces `INTEGRITY_MISMATCH` or `BROKEN`
  and exit `5`
- later source/model changes produce separate advisories without changing
  historical artifact integrity
- re-observation creates a new run and preserves the old one
- documentation states the trusted-local threat model and local-hash limits
- no signing, ledger, sandbox, ACL system, quarantine database, repair,
  deletion, service, UI, or downstream integration is introduced
- a fresh reviewer returns `PASS` under this amended contract

An observation’s artifact integrity is self-contained in its run directory
and bundled schema contract. The source, lockfile, and model inventory are
recorded provenance; current source/model comparisons are optional advisories.

### 13.12 Continuation sequence

1. Commit this accepted amendment and update `CURRENT.md`.
2. Update schemas and terminology first.
3. Implement runtime preflight, private source snapshots, and PDF model
   pre/post validation.
4. Implement `tcw verify` without extractor imports.
5. Add the required amendment tests and documentation.
6. Run all existing fast/full checks plus manual `observe` and `verify`.
7. Run a fresh milestone review against the amended contract.
8. Repeat builder/reviewer passes until `PASS`.

A finding based solely on malicious same-user mutation deliberately timed
after final staged verification is outside the accepted threat model and is
non-blocking. Ordinary correctness, schema, concurrency, source snapshot,
model stability, and verifier findings remain blocking.

### 13.13 Authenticity boundary

This clarification was accepted on 2026-07-21. Authenticity is deliberately
not a fourth v0.1 `VERIFIED` promise. v0.1 adds no observation fingerprint,
signature, key management, identity, timestamp authority, external anchor, or
placeholder verification state for those capabilities.

If a future concrete user story introduces an external relying party,
disputed evidence, or a chain-of-custody requirement, a separate additive
verification category may be designed then. It must not be inferred from the
v0.1 artifact-integrity state or retroactively broaden `VERIFIED`.

Review remains blocking for invalid timestamp syntax, multiline/unsanitized
errors, Docling schema identity that disagrees with `document.json`, or any
other failed structural or enumerated semantic check. Review is non-blocking
when its sole claim is that v0.1 cannot recover the historically original
value of schema-valid, non-derivable metadata without an external trust
anchor.
