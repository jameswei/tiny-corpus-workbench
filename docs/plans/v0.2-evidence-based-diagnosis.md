# Milestone Plan: v0.2 — Evidence-Based Diagnosis

**Status:** Accepted by the owner on 2026-07-23

## Outcome and boundaries

Add a local, offline diagnosis layer that consumes one structurally verified
v0.1 observation and analyzes only its canonical `DoclingDocument`. Publish a
separate application-immutable diagnosis; never modify the source, observation,
or extraction artifacts.

Deliver eight fixed deterministic rules, stable evidence-backed findings,
`findings.json`, deterministic `report.md`, diagnosis verification, hybrid
diagnostic fixtures, documentation, learning material, CI coverage, and release
`v0.2.0`.

MarkItDown output remains descriptive and cannot produce findings. Diagnosis
does not authorize repair. Refinement, approvals, revisions, semantic
correctness, extractor ranking, conflicting-revision analysis, batch execution,
services, UI, and downstream RAG work remain out of scope.

## Public interfaces and publication

Add:

```text
tcw diagnose OBSERVATION_DIRECTORY
  [--output-root build/evidence-based-diagnosis]

tcw verify-diagnosis DIAGNOSIS_DIRECTORY
  [--observation OBSERVATION_DIRECTORY]
```

`diagnose` accepts only an intact `tcw.preparation-manifest/v0.1` observation
whose Docling result is `SUCCESS` or `PARTIAL_SUCCESS` and contains usable
canonical JSON. It runs all rules; v0.2 has no rule-selection or threshold
configuration.

Publish atomically without overwrite:

```text
<output-root>/<source-key>/<observation-run-id>/<diagnosis-run-id>/
  diagnosis-manifest.json
  findings.json
  report.md
```

Snapshot the observation's complete file/directory inventory before analysis
and recheck it immediately before publication. Any path, kind, size, hash,
identity, symlink, or inventory change aborts publication.

Successful diagnosis prints one sorted compact JSON line containing
`diagnosis_id`, `finding_count`, `manifest`, `run_id`, and `status`. Use exits:
`0` completed diagnosis, `2` invalid or unsupported input, `4` canonical
Docling artifact unavailable, `5` integrity change or publication conflict,
`6` incompatible locked runtime/schema API, and `1` unexpected failure.
Failures write no stdout and sanitize application diagnostics.

`verify-diagnosis` validates the diagnosis independently. With `--observation`,
it compares the supplied observation identity and reruns all rules only when it
matches. Its result reports:

```text
artifact_integrity: VERIFIED | INTEGRITY_MISMATCH | BROKEN
observation_state: NOT_CHECKED | MATCH | CHANGED | MISSING | ERROR
derivation_state: NOT_CHECKED | MATCH | MISMATCH | ERROR
```

Advisory observation/derivation states do not change exit `0` for an intact
diagnosis. Integrity mismatch or broken content exits `5`; retain exits `2`,
`6`, and `1`. Neither command mutates its input.

Keep `tcw observe` and `tcw verify` behavior unchanged.

## Schemas and deterministic identity

Add closed Draft 2020-12 schemas:

- `tcw.diagnosis-manifest/v0.2`
- `tcw.finding-set/v0.2`
- `tcw.diagnosis-verification-result/v0.2`

The manifest records run/diagnosis identity, creation time, `FINDINGS` or
`NO_FINDINGS`, source and observation identity, manifest and
canonical-document hashes/sizes, Docling schema identity, CPython/lock/package
versions, complete ruleset provenance and parameter hash, counts by
severity/rule, and immutable descriptors for findings/report.

`findings.json` records the diagnosis and observation identities, canonical
artifact/hash, complete ruleset descriptor, summary counts, and sorted
findings. Each finding contains exactly:

```text
finding_id
rule_id
rule_version
severity
summary
document_refs
evidence
```

Canonical references use Docling paths such as `#/body`, `#/texts/7`, and
`#/tables/0`. Table-cell evidence uses the table reference plus zero-based
row/column coordinates. Sort and deduplicate references. Sort evidence by
stable name; values are scalars or scalar lists. Do not copy arbitrary source
passages—record counts, offsets, coordinates, page numbers, hashes, and
relationship kinds.

Compute full lowercase SHA-256 identifiers:

- `diagnosis_id`: canonical observation ID, observation-manifest hash,
  canonical-document hash, and canonical ruleset descriptor.
- `finding_id`: diagnosis ID, rule ID/version, sorted references, and canonical
  evidence.

`findings.json` and `report.md` are byte-identical for repeated diagnosis of
the same observation/ruleset. The manifest may contain the unique run ID and
timestamp. The report contains no timestamps or absolute paths and states that
diagnosis neither authorizes mutation nor certifies overall quality.

## Fixed rules

Normalize specified text with Unicode NFC, normalized line endings, collapsed
Unicode whitespace, and outer whitespace removal. Preserve case.

| Rule | Severity | Deterministic condition |
| --- | --- | --- |
| `TCW-D001 EMPTY_DOCUMENT` | `ERROR` | Body has no non-whitespace text-item or table-cell content. Emit at `#/body`. |
| `TCW-D002 SUSPICIOUSLY_SHORT_DOCUMENT` | `INFO` | Body has 1–199 non-whitespace characters across text items and table cells. Suppress if D001 fires. |
| `TCW-D003 REPLACEMENT_CHARACTER` | `ERROR` | U+FFFD occurs in a text item or table cell. Emit per affected item; use zero-based Unicode code-point offsets or cell coordinates. |
| `TCW-D004 DUPLICATE_TEXT_BLOCK` | `WARNING` | Identical case-sensitive normalized BODY `text`/`paragraph` blocks of at least 80 characters occur in two or more items. Emit once per group with refs, normalized-text hash, length, and count. |
| `TCW-D005 HEADING_LEVEL_JUMP` | `WARNING` | First BODY section heading is deeper than level 1, or a later heading increases by more than one from the previous heading. Emit on the later heading with previous/current levels. |
| `TCW-D006 ORPHAN_CAPTION` | `WARNING` | A caption has no valid incoming table/picture `captions[]` reference, or a declared reference is unresolved or targets a non-caption item. Emit once per orphan caption or invalid declared reference. A table/picture with no declared caption is not itself a finding. |
| `TCW-D007 REPEATED_PAGE_MARGIN_TEXT` | `WARNING` | PDF only: identical normalized BODY text of 3–200 characters occurs in the same top or bottom outer 10% on at least three distinct pages. Convert boxes to top-left origin, normalize by page size, and classify midpoint `<=0.10` as top and `>=0.90` as bottom. Group separately by normalized-text hash and band. Exclude furniture. |
| `TCW-D008 MISSING_PDF_PROVENANCE` | `WARNING` | PDF only: any Docling text, table, or picture item in any content layer has no provenance entries. Emit once per item. |

Run rules in rule-ID order. Rules are pure, read-only, offline functions.

## Implementation and data flow

- Load diagnosis and diagnosis-verification runtimes lazily.
- Reuse v0.1 verification to establish input integrity, then privately snapshot
  manifest and canonical JSON bytes.
- Reload through pinned `DoclingDocument` and build a read-only context
  containing source media type, body reading order, item/ref indexes, reverse
  caption links, table cells, page geometry, provenance, and canonical hashes.
- Run the rules, canonicalize findings, compute IDs/summaries, validate
  findings, render deterministic Markdown, then construct and validate the
  manifest.
- Validate staged paths, schemas, hashes, identities, summaries, references,
  expected inventory, and absence of symlinks before exclusive atomic rename.
- Generalize v0.1 publication helpers only if released v0.1 behavior remains
  unchanged; otherwise use diagnosis-specific wrappers.
- Optional verification reruns diagnosis from the matching observation
  snapshot and requires byte equality for `findings.json` and `report.md`.
- Set project version to `0.2.0`; diagnosis requires no model files or network
  access.

## Fixtures and tests

Keep `fixtures/golden/` exactly unchanged. Add a separate versioned CC0
diagnosis corpus and deterministic generator/registry:

- raw `short-note.md` producing only D002;
- raw `structural-traps.md` producing D003, D004, and D005;
- deterministic three-page PDF producing D007, with unique body text and a
  furniture-layer negative case;
- schema-valid constructed `DoclingDocument` cases for D001, D006, and D008;
- boundary/negative constructed cases for every rule.

Required coverage:

- Every rule's positive, negative, boundary, suppression, grouping, and
  ordering behavior.
- Stable/changing diagnosis and finding identifiers.
- Closed-schema rejection and semantic cross-checks.
- All twelve released v0.1 observations diagnose to `NO_FINDINGS`.
- Raw diagnostic cases pass `observe → diagnose → verify-diagnosis` offline.
- Repeated runs produce byte-identical findings/reports.
- Valid partial-success observations are accepted; missing canonical JSON is
  rejected without publication.
- Invalid, corrupt, changed, symlinked, or concurrently modified inputs never
  publish.
- Atomic no-overwrite, concurrent publication, staged corruption, unexpected
  content, and cleanup behavior.
- Verifier detection of missing, added, malformed, unsafe, inconsistent, or
  replaced diagnosis content.
- All optional observation and derivation states.
- Input bytes, modes, and mtimes remain unchanged.
- CLI stdout/stderr, exit codes, sanitization, lazy imports, and network/model
  independence.
- Deterministic generation, checkout portability, compilation, site
  validation, and diff hygiene.

Extend existing fast CI with unit/schema/fixture/static checks and full CI with
extraction-plus-diagnosis integration. A fresh Ubuntu CPython 3.12 checkout must
pass both jobs.

## Accepted assumptions

- Canonical Docling JSON and v0.1 source metadata are the only diagnostic
  inputs.
- The eight rule definitions, severities, and thresholds are fixed for v0.2.
- Diagnosis publications are derived immutable records linked by hashes.
- Local hashes provide tamper evidence under the existing trusted-local model,
  not authenticity or signatures.
- Findings do not approve changes; controlled revisions begin in v0.3.
