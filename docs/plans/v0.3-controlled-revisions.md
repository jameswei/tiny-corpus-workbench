## Interfaces and contracts

Add:

```text
tcw draft-refinement DIAGNOSIS_DIRECTORY
  --finding FINDING_ID
  --base BASE_DIRECTORY
  --output DECISION_FILE

tcw resolve-refinement DECISION_FILE
  --diagnosis DIAGNOSIS_DIRECTORY
  --base BASE_DIRECTORY
  [--output-root build/controlled-revisions]

tcw verify-refinement REFINEMENT_DIRECTORY
  [--diagnosis DIAGNOSIS_DIRECTORY]
  [--base BASE_DIRECTORY]
```

- One draft targets one finding. Only `decision`, `decided_by`, and an
  optional note are editable.
- Resolution requires `APPROVED` or `REJECTED`; `PENDING` is invalid.
- Approval creates one immutable successor revision. Rejection publishes an
  immutable decision record without a prepared document.
- Re-diagnose each approved revision before drafting its successor.
- Extend `tcw diagnose DOCUMENT_DIRECTORY` to accept a verified observation
  or approved v0.3 revision. Add `--subject` to `verify-diagnosis` while
  retaining v0.2 `--observation` compatibility.
- Add closed v0.3 schemas for diagnosis, findings, refinement drafts and
  manifests, transformation history, and verification. Existing v0.2
  diagnoses must remain verifiable.

The v0.3 diagnosis subject is either:

- an intact `tcw.preparation-manifest/v0.1` observation, identified by its
  `observation_id`; or
- an intact applied v0.3 refinement, identified by its `revision_id` and
  linked to its originating observation.

Record the subject kind, subject ID, canonical document path, size and hash,
and originating observation identity. Compute the v0.3 diagnosis ID from that
subject descriptor, the canonical document hash, and the complete v0.3
ruleset. Publish v0.3 diagnoses below
`<output-root>/<source-key>/<subject-id>/<diagnosis-run-id>/`. Dispatch
verification by schema version: v0.2 verification and rule reruns retain their
released observation-only contract, while v0.3 verification uses `--subject`.

## Diagnosis and refinement behavior

Add:

- `TCW-D009 NORMALIZABLE_WHITESPACE` (`INFO`): inspect text-bearing items in
  the body content layer and cells in body-layer tables. Exclude items labeled
  `code` or `formula`. Normalize CRLF and CR to LF. Within each line, map every
  Unicode whitespace character other than CR or LF to ASCII space, collapse
  each horizontal run to one space, and remove leading and trailing horizontal
  space. Preserve the number and order of internal line breaks. Emit one
  finding per item or table cell when the normalized value differs. Group all
  changed maximal whitespace spans in that finding and record their starting
  Unicode code-point offsets, count, original-text hash, and normalized-text
  hash. Table-cell evidence also records its zero-based row and column.
- `TCW-D010 POSSIBLE_LINE_END_HYPHENATION` (`WARNING`): inspect the same
  eligible items and cells. Match an ASCII hyphen whose maximal left and right
  Unicode-alphabetic fragments each contain at least two code points. Between
  the hyphen and right fragment, allow horizontal Unicode whitespace, exactly
  one logical line break (`LF`, `CR`, or `CRLF`), and more horizontal
  whitespace. Reject blank-line boundaries, and require the first right-side
  character to be lowercase. Emit one finding per item or table cell, grouping
  all matches. Record the hyphen code-point offsets, occurrence count,
  original-text hash, and repaired-text hash, plus row and column for table
  cells. Repair each match by deleting the hyphen and all intervening
  whitespace so the two alphabetic fragments join.

Use these fixed mappings:

| Finding | Refiner | Behavior |
| --- | --- | --- |
| D009 | `TCW-R001 WHITESPACE_NORMALIZATION` | Normalize line endings and horizontal whitespace in prepared text. |
| D007 | `TCW-R002 REPEATED_BOILERPLATE_REMOVAL` | Move repeated margin items from body to furniture while retaining evidence. |
| D010 | `TCW-R003 DETERMINISTIC_DEHYPHENATION` | Remove the approved hyphen, adjacent whitespace, and single line break. |

- Change `text`, never `orig`. Preserve provenance and stable references.
- Each approved publication contains prepared Docling JSON, derived Markdown,
  the finalized decision, one transformation, append-only history, a
  deterministic report, and a manifest.
- Record the finding, actor label, refiner version, affected refs, reversible
  edit data, and whole-document before and after hashes.
- Verification must replay both the forward operation and its inverse. The
  inverse must reproduce the base `document.json` byte-for-byte.
- Snapshot and recheck every input before exclusive atomic publication. Never
  overwrite or publish inside an input directory.
- Retain exits `0`, `1`, `2`, `4`, `5`, and `6`. Rejection is a successful
  recorded outcome.

The draft is a closed `tcw.refinement-draft/v0.3` document with an immutable
proposal and an editable decision. The proposal records `REQUESTED`, the base
and diagnosis identities, the finding, fixed refiner descriptor, affected
references, deterministic forward and inverse edits, and a stable `draft_id`.
The decision initially contains `PENDING`, a null `decided_by`, and an optional
note. `resolve-refinement` recomputes the proposal and rejects any modification
outside those three editable decision fields.

Publish a resolved record below
`<output-root>/<source-key>/<origin-observation-id>/<refinement-run-id>/`:

```text
refinement-manifest.json
decision.json
report.md
transformation.json          # APPLIED only
history.json                 # APPLIED only
prepared/document.json       # APPLIED only
prepared/document.md         # APPLIED only
```

The manifest status is `APPLIED` for an approved decision and `REJECTED` for a
rejected decision. An applied transformation records state `APPLIED`, the
parent identity and document hash, finding and decision identity, refiner
version, exact forward and inverse edit data, and prepared-document hash. An
applied child copies its verified parent history and appends exactly one
transformation. A rejected record has no `revision_id`, transformation,
history, or prepared artifacts.

Add these closed Draft 2020-12 contracts:

- `tcw.diagnosis-manifest/v0.3`
- `tcw.finding-set/v0.3`
- `tcw.refinement-draft/v0.3`
- `tcw.refinement-manifest/v0.3`
- `tcw.transformation-history/v0.3`
- `tcw.refinement-verification-result/v0.3`

Refinement verification reports artifact integrity plus diagnosis, base,
derivation, and reversibility states. Without optional inputs, external states
are `NOT_CHECKED`; rejected records use `NOT_APPLICABLE` for derivation and
reversibility. With matching inputs, replay the forward edit to reproduce the
prepared bytes and the inverse edit to reproduce the base bytes.
